## About the Workflow

1. Plans are prepared in `plans/NN.<slug>.md` (generated by `/plan`)
2. Plans receive an appended review section before implementation begins
3. Execute with `/dev NN` — read the plan, implement, then generate `reports/NN.<slug>.md` and update `STATUS.md`
4. Reports may later receive appended review sections.

At the beginning of every session, read in order:

1. This file (`.claude/CLAUDE.md`) — conventions
2. `STATUS.md` — current progress and open review findings
3. The relevant plan in `plans/` if executing `/dev` (including any appended reviews)

## About the Python Environment

Use `uv` for all environment and package management. **Never use `pip install` directly.**

```bash
# Virtual environment (do NOT recreate)
# Created with: uv venv --python=3.12
# Activate before ANY Python execution
source .venv/bin/activate

# Install packages
uv pip install <package>
```

- Always activate the venv before running Python scripts, tests, or installing packages.
- If a required package is missing, install it with `uv pip install` and note it in the current report under **Environment Changes**.
- Do not modify `pyproject.toml` or `requirements.txt` without an explicit instruction in the plan.

## About the Costomized Commands

1. `/plan <task>`: e.g. `/plan add ablation for X`. Generate `plans/NN.<slug>.md` after a grounding pass on `STATUS.md`, recent reports, and target files.
2. `/dev <plan-id>`: e.g. `/dev 03`. Execute the plan, write `reports/NN.<slug>.md`, update `STATUS.md`.

## About the Code Conventions

- Type hints on all function signatures; Google-style docstrings on public APIs.
- `pathlib.Path` over `os.path`; `logging` over `print()` in library code.
- Match existing patterns rather than introducing new ones.
- Tests in `tests/` mirror `src/`; run `python -m pytest tests/` after changes.
- If tests fail, fix before writing the report: a `/dev` with failing tests must report this as a blocker, not a success.

## About the Git

- All git operations are handled by the user. Never run git commands.

## About the Guardrails

- Always stay in scope. The plan's Out of Scope section is binding. Out-of-scope work goes in the report's Follow-ups, never silently into the implementation.
- Do not self-review. Write reports factually — list what was done and what is uncertain. Judgment is handled separately.
- When facing a design decision the plan didn't anticipate, document it in the report's Decisions section rather than pausing — unless it invalidates the plan, in which case stop and ask.
